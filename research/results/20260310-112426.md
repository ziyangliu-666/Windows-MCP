# Reliability Results Log

Timestamp: 20260310-112426

## Session Goal
- Inspect the current Windows-MCP implementation.
- Build the first research artifacts.
- Run three simple benchmark tasks before broad code changes.

## Architecture Findings So Far
- Single-module MCP tool registration in `src/windows_mcp/__main__.py`.
- Core desktop state is UIA tree + window enumeration + optional screenshot/DOM.
- Label-based actions depend on cached `desktop_state`.
- Post-action verification is mostly absent in tool implementations.
- Existing retries focus on auth and tree capture, not end-to-end task success.

## Findings
- The live tool server is strongest on non-UI tasks (`Clipboard`, `FileSystem`).
- Native desktop app discovery is brittle across localized app names.
- Success strings from `App.launch` and `App.switch` are not trustworthy without an immediate re-observation step.
- The keyboard input path contains a source-level defect (`_INPUTUnion` import bug).
- Current `Snapshot` output is insufficient for verifying some UI tasks because informative text is not surfaced.

## Patch Candidates
- Explicit `_INPUTUnion` import fix in `src/windows_mcp/uia/core.py`.
- Locale-aware app launch fallback using Start App IDs or aliases.
- Postcondition verification inside `App.launch` and `App.switch`.
- Richer `Snapshot` output for informative/text nodes.
