### 4. Hot-Reload Shell Boundary Check
- Preconditions:
  - Source patch added new shell-side health fields.
- Actions:
  - `DevServer(mode="reload")`
  - `DevServer(mode="health")`
- Postcondition:
  - Fail.
- Expected:
  - Health payload should expose the new source-identity fields.
- Actual:
  - Worker generation incremented, but health payload stayed on the old schema.
- Classification:
  - Hot reload refreshes the worker only; shell-owned code remains stale until shell restart and host reconnect.

### 5. Wait Tool Long-Call Diagnosis
- Live results:
  - `Wait(2)` passed.
  - `Wait(4)` passed.
  - `Wait(8)` failed with outer tool deadline.
  - `Wait(11)` failed with outer tool deadline.
- Local control results:
  - In-memory `build_local_mcp()` `Wait(8)` passed.
  - Direct worker HTTP `_call_worker_tool(url, "Wait", {"duration": 8})` timed out under an explicit 20 second guard.
  - Local hot-shell `Wait(8)` passed after patching the shell to execute `Wait` locally.
- Classification:
  - The defect is in hot mode's internal worker HTTP path for longer-running forwarded calls, not in the core `Wait` handler itself.

## Reverted Experiment
- A first attempt to switch the hot supervisor's internal worker bridge from HTTP to stdio was reverted in the same session.
- Why it was reverted:
  - direct stdio worker calls were healthy;
  - but supervisor-managed forwarded calls failed after the metadata handshake with `Server session was closed unexpectedly`.
- Result:
  - hot mode remains on the known-good HTTP worker bridge for now;
  - stdio stays a strong architecture candidate, but only with a persistent-session design rather than reconnect-per-call reuse.
- Follow-up evidence:
  - a persistent stdio client session successfully handled `__wmcp_worker_meta`, `Wait(8)`, and `Clipboard(get)` against the same worker in one connection.
